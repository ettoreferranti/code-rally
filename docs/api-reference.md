# CodeRally API Reference

CodeRally exposes a small REST surface for lobby/bot/user management
plus a single WebSocket endpoint for real-time game play.

> **Auto-generated docs**: FastAPI publishes the full OpenAPI schema at
> [`/docs`](http://localhost:8000/docs) (Swagger UI) and `/redoc`
> while the dev server is running. That's the authoritative reference
> for parameters and response shapes. This document is a curated
> high-level overview and the only spec for the WebSocket protocol
> (which OpenAPI doesn't cover).

## Base URL

```
Development: http://localhost:8000
```

## Authentication

There is **no authentication** today. Users are identified by their
typed username (registered idempotently via `POST /users/register`).
This is a local dev / research project.

## Health

```
GET /health         → 200 {"status": "healthy", "database": "connected"}
GET /health/ready   → 200 {"status": "ready"}
```

## Configuration

```
GET /config/physics
```

Returns the server-side physics constants (MAX_SPEED, ACCELERATION,
TURN_RATE, GRIP_*, etc.) so the frontend can stay in sync.

## Tracks

```
POST /tracks/generate
  Body: {"seed": number?, "difficulty": "easy"|"medium"|"hard"|"extreme"}
```

Most clients don't call this directly — tracks are generated by the
backend when a lobby's host starts the race. See `core/track.py` for
the generator and `Track` schema.

## Users

```
GET  /users                        # List all users
POST /users/register               # Idempotent registration
  Body: {"username": "alice"}
GET  /users/{username}             # Get user by name
DELETE /users/{username}           # Delete user + all their bots
```

## Bots

The unified bot library (Python + LLM in the same table).

```
GET  /bots/users/{username}/bots   # List a user's bots (no code in payload)
GET  /bots/{id}                    # Get a single bot (includes code)
POST /bots/users/{username}/bots   # Create
PUT  /bots/{id}                    # Update
DELETE /bots/{id}                  # Delete

GET  /bots/templates               # Curated Python bot templates
GET  /bots/templates/{id}          # Template source code
```

**Create bot — Python**:
```json
POST /bots/users/alice/bots
{
  "name": "SpeedBot",
  "kind": "python",
  "code": "class SpeedBot(BaseBot):\n    def on_tick(self, state): ..."
}
```

**Create bot — LLM**:
```json
POST /bots/users/alice/bots
{
  "name": "AggressiveQwen",
  "kind": "llm",
  "model_path": "mlx-community/Qwen2.5-1.5B-Instruct-4bit",
  "system_prompt": "You are a rally driver racing AS FAST AS POSSIBLE..."
}
```

**Update bot** accepts the same fields as Create (all optional). The
kind is **immutable** after creation. Updating `code` on an LLM bot,
or `model_path` on a Python bot, returns 400.

## LLM Model Presets

```
GET /llm-models
```

Returns the curated MLX preset list that the Tinker UI populates the
LLM-bot model dropdown from. Edit
`backend/app/agents/llm_model_presets.json` to add/remove entries
without touching code; restart the server to pick up changes.

## Lobbies

Lobbies are in-memory only. They don't persist across server restarts.

```
GET  /lobbies?status=waiting|racing|finished       # List lobbies
POST /lobbies                                       # Create
  Body: {
    "name": "Rally Night",
    "host_player_id": "alice",
    "track_difficulty": "medium",
    "max_players": 8
  }
GET  /lobbies/{id}                                  # Get lobby state
PUT  /lobbies/{id}/settings?player_id=alice         # Host updates settings
DELETE /lobbies/{id}?player_id=alice                # Disband (host OR creator)
```

The `LobbyListItem` payload includes `host_player_id` (current host,
can transfer between humans) AND `creator_player_id` (original
creator, stable identity used to label "YOUR LOBBY" rows). See
`architecture.md` for the rationale.

## Game WebSocket

```
WS /game/ws?lobby_id=<id>&player_id=<name>          # Join lobby
WS /game/ws?lobby_id=<id>&player_id=<name>&spectate=true   # Spectate lobby
WS /game/ws?session_id=<id>&player_id=<name>         # Connect to active race
WS /game/ws?session_id=<id>&player_id=<name>&spectate=true # Spectate race
```

### Client → Server messages

```json
{"type": "leave_lobby"}
{"type": "start_race"}
{"type": "add_bot_to_lobby", "data": {"bot_id": 42}}
{"type": "input", "data": {
  "accelerate": true, "brake": false,
  "turn_left": false, "turn_right": true,
  "nitro": false
}}
{"type": "pong"}
```

The mid-race `submit_bot` message and the LLM-specific
`add_llm_bot_to_lobby` message were removed when the Tinker / lobby
flow was unified — all bots are added pre-race via `add_bot_to_lobby`
with a `bot_id` from the user's library.

### Server → Client messages

```json

{"type": "lobby_joined", "data": {"player_id": "alice", "lobby": {…}}}

// Spectator equivalent
```

## Coordinate System

All positions / vectors in the WS payload are in **engine units**
in a **y-DOWN** coordinate system (canvas convention). The frontend
displays speeds in km/h via the same conversion the LLM agent layer
uses: `velocity_magnitude * 0.6` (so MAX_SPEED of 300 units/s ≈ 180
km/h).

## Error Responses

Standard FastAPI error shape:

```json
{"detail": "Only the lobby's host or creator can disband it"}
```

Common codes:
- **400** — validation / business-rule failure (e.g. duplicate bot
  name, missing required kind-specific field).
- **403** — host-only action attempted by non-host.
- **404** — lobby / user / bot not found.
- **500** — unexpected server error (check logs).

For the WebSocket, errors come as `{"type": "error", "data": {…}}`
messages on the same connection rather than HTTP status codes.
